📋 PyWebFW — Software Requirements Specification

Last updated: 2026-06-14 | Version: 0.2.0 (Open-Core) | 5-Cycle Audit Complete ✅

1. Overview

PyWebFW is an OOP-first, reusable Python web framework providing:

  • Public front-end + CMS with dynamic routes
  • Admin area (14 screens) with form editors
  • Public + Admin APIs with RBAC
  • Scheduler engine (8 jobs) + event bus
  • Auth: revocable tokens, 2FA TOTP, forced password change
  • Plugin system (AppModule) for extensibility

2. Core Features (Community Edition)

2.1 Authentication & Authorization

  • PBKDF2-310k password hashing
  • HMAC-signed tokens with instant revocation (token_version)
  • Forced password change on first login
  • 2FA TOTP (RFC 6238) — setup/enable/disable
  • Login throttle: 3/3min per IP (hardened from 5/5min)
  • RBAC: admin, editor, viewer roles
  • Session revocation on logout + password change

2.2 Content Management

  • Dynamic content routes: /{slug} catch-all
  • FTS5 full-text search (with LIKE fallback)
  • CRUD: create/read/update/delete published content
  • Auto-301 redirect when slug changes (via event bus)
  • JSON-LD NewsArticle schema (auto on publish)

2.3 Admin Features

  • Form editors for Users (username/email/password/role/active)
  • Form editors for Menus (title/url/area/position)
  • Pagination (prev/next) on all data tables
  • Logs: audit trail for every mutation (who/what/when/level)
  • Contact manager: inbox with read/delete
  • Dashboard: metrics cards, DB health, recent activity
  • Settings: runtime key-value config + maintenance mode

2.4 Security Hardening ✅ Hardened (5-cycle audit)

  • debug=False by default (was True)
  • token_ttl=3600s (1 hour, was 86400s = 1 day)
  • HTTPS via Caddy with auto Let's Encrypt + HSTS ✅
  • Security headers: CSP (per-request nonce), X-Frame-Options, Referrer-Policy ✅
  • HttpOnly/SameSite=Strict cookies ✅
  • DB connection pool: 20 (was 5)
  • Idle timeout: 300s (was 3600s) — free stale connections
  • Query timeout: 30s (prevents hanging)
  • Audit logging: IP + UA on login attempts

2.5 Operations & Monitoring

  • Health endpoint (/healthz): uptime, DB latency, pool status
  • Scheduler monitor: 8 jobs with status/duration/audit
  • Traffic analytics: page views (7-day), online users (5min window)
  • robots.txt for SEO crawl control
  • Graceful error pages (404/500 styled)

4. Deployment Architecture

  • Docker 3-mode split: public (FE), admin (APIs), scheduler (jobs)
  • Caddy reverse proxy with auto TLS + HSTS
  • SQLite (default) + PostgreSQL (Pro)
  • In-memory cache (default) + Redis (Pro)
  • PowerShell deploy scripts: local / SSH / multi-container

5. Extension (Plugin System)

All extensibility via AppModule contract:

  • register_services() — add to DI container
  • init_schema() — create tables (idempotent)
  • controllers() — mount web/API pages
  • jobs() — register scheduled tasks
  • subscribe_events() — react to domain events

6. Testing

69 end-to-end tests through the real HTTP stack (pytest):

  • Security: login throttle, cookies, token revocation, password change
  • CMS: CRUD, search, dynamic routes, auto-redirect
  • 2FA TOTP: setup, enable, login flow
  • Admin forms: user/menu editors, pagination
  • Audits: SEO, security headers, DB optimization
